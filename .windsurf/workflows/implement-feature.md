# /implement-feature — Feature Implementation Workflow

> Uses the **feature-implementer** skill to implement a feature whose planning artifacts
> were generated by the `/breakdown-initiative` workflow.

## Pre-requisites

- The `/breakdown-initiative` workflow has been run for the parent epic.
- The feature directory exists with all three documents:
  - `prd.md`, `implementation-plan.md`, `test-strategy.md`

## Workflow Steps

### Step 1 — Collect Input

Ask the user for:

| Input | Description | Example |
|-------|-------------|---------|
| **Epic name** | Directory name of the epic | `multi-channel-notifications` |
| **Feature name** | Directory name of the feature | `f1-push-notification-channel` |
| **Execution mode** | `phased` (pause per phase) or `full` (all at once) | `phased` |

### Step 2 — Validate Artifacts Exist

// turbo

Verify these files exist before proceeding:

```
/docs/ways-of-work/plan/{epic-name}/epic.md
/docs/ways-of-work/plan/{epic-name}/arch.md
/docs/ways-of-work/plan/{epic-name}/project-plan.md
/docs/ways-of-work/plan/{epic-name}/{feature-name}/prd.md
/docs/ways-of-work/plan/{epic-name}/{feature-name}/implementation-plan.md
/docs/ways-of-work/plan/{epic-name}/{feature-name}/test-strategy.md
```

If any are missing, notify the user and suggest running `/breakdown-initiative` first.

### Step 3 — Check Feature Dependencies

// turbo

Read `/docs/ways-of-work/plan/{epic-name}/project-plan.md` and find the dependency table for this feature.

If this feature has dependencies (e.g., F3 depends on F1):
- Verify the dependent feature has already been implemented in the codebase.
- If not, warn the user and ask whether to proceed anyway or implement the prerequisite first.

### Step 4 — Load Context Documents

// turbo

Read the 6 input documents in this order:
1. `../epic.md` — business context
2. `../arch.md` — system architecture
3. `../project-plan.md` — dependencies and sprint plan
4. `prd.md` — feature requirements
5. `implementation-plan.md` — phases and file paths
6. `test-strategy.md` — test cases and quality gates

### Step 5 — Execute Feature Implementer Skill

// turbo

1. Read the full SKILL.md at `/feature-implementer/SKILL.md`.
2. Execute the skill following its **Execution Protocol** phase by phase.
3. For each phase:
   a. **Understand** — identify all artifacts to create/modify
   b. **Implement** — write full file contents (no TODO stubs)
   c. **Verify** — map to acceptance criteria, run `./mvnw compile`
   d. **Write Tests** — implement test cases from the Test Strategy
4. If `execution_mode` is `phased`:
   - After each phase, notify the user with `PathsToReview` listing all files created/modified.
   - Wait for user confirmation before proceeding to the next phase.
5. If `execution_mode` is `full`:
   - Run all phases sequentially without pausing.

### Step 6 — Quality Gate

// turbo

After all phases are complete:

1. Run `./mvnw compile` to verify all code compiles.
2. Run `./mvnw ktlint:format` to fix formatting.
3. Run `./mvnw test` to ensure all tests pass.
4. Produce the **Implementation Summary** checklist (as defined in the skill).

### Step 7 — Notify User

// turbo

Present the Implementation Summary to the user with `PathsToReview` listing:
- All new files created
- All existing files modified
- Any remaining items not yet implemented (with reasons)

## Error Handling

- If `./mvnw compile` fails during a phase, fix the errors before proceeding.
- If a test fails, investigate and fix. If the fix requires changing the implementation,
  stay in the current phase and re-verify.
- If a prerequisite feature is missing, offer to implement it first using this same workflow.
